# Testing Embedded Code

Testing embedded code is harder than testing desktop software because the hardware may not exist yet, may be expensive, or may be in another country. The solution is to write code that can be tested without hardware — and to verify hardware-specific behavior separately.

---

## The Testing Pyramid

```
                  ┌───────────┐
                  │  HIL Test │  Real hardware, real environment
                  │  (few)    │  Slow, expensive, catches integration bugs
                  ├───────────┤
                  │Integration│  Multiple components, may use simulators
                  │  Tests    │
                  ├───────────┤
                  │   Unit    │  Individual functions, host-compiled
                  │   Tests   │  Fast, cheap, catches logic bugs
                  └───────────┘  (most)
```

Write most tests as unit tests that run on your Linux workstation. Run them in CI. Reserve HIL tests for hardware-dependent behavior.

## Unit Testing with Unity + CMock

**Unity** is a lightweight C unit test framework used heavily in embedded and flight software.

```c
// test_adcs.c
#include "unity.h"
#include "adcs.h"
#include "mock_imu.h"    // generated by CMock from imu.h

void setUp(void) {
    adcs_init();
}

void tearDown(void) {}

void test_quat_normalize_unit_vector(void) {
    Quaternion_t q = {0.1, 0.2, 0.3, 0.4};  // not unit
    quat_normalize(&q);

    double norm = sqrt(q.q1*q.q1 + q.q2*q.q2 + q.q3*q.q3 + q.q4*q.q4);
    TEST_ASSERT_DOUBLE_WITHIN(1e-10, 1.0, norm);
}

void test_adcs_sensor_failure_escalates_to_fdir(void) {
    // Mock: IMU returns failure
    imu_read_gyro_ExpectAndReturn(NULL, false);
    imu_read_gyro_IgnoreArg_rate();

    // Expect: FDIR receives anomaly report
    fdir_report_anomaly_Expect(ANOMALY_IMU_FAILURE, 0);

    SensorMeasurement_t meas;
    bool result = adcs_read_sensors(&meas);

    TEST_ASSERT_FALSE(result);
}

int main(void) {
    UNITY_BEGIN();
    RUN_TEST(test_quat_normalize_unit_vector);
    RUN_TEST(test_adcs_sensor_failure_escalates_to_fdir);
    return UNITY_END();
}
```

**CMock** auto-generates mock implementations from header files:
```bash
ruby cmock/lib/cmock.rb --mock_path=mocks src/imu.h
# Generates mocks/mock_imu.c and mocks/mock_imu.h
# Provides: imu_read_gyro_Expect(), imu_read_gyro_ExpectAndReturn(), etc.
```

## Abstracting Hardware for Testability

The key pattern: never call hardware directly from application logic. Put hardware access behind an interface.

```c
// BAD — application directly calls hardware register
void read_temperature(float* out) {
    while (!(ADC1->SR & ADC_SR_EOC));   // wait for conversion — untestable
    *out = ADC1->DR * 0.1f - 40.0f;
}

// GOOD — hardware access hidden behind a function pointer / interface
typedef bool (*temp_read_fn)(float* out);

static temp_read_fn g_temp_reader = adc_read_temperature;  // real hardware

// In unit test
static bool mock_temp_reader(float* out) {
    *out = 25.0f;   // return fake value
    return true;
}

// Test can inject the mock
void set_temp_reader(temp_read_fn fn) { g_temp_reader = fn; }
```

On NASA cFS, the PSP and OSAL serve this role — application code never calls RTOS or hardware functions directly, so the same app runs on RTEMS (target) and Linux (test).

## Renode — Full System Simulation

Renode is an open-source hardware simulator that can run your unmodified firmware binary without any physical hardware.

```bash
# Install Renode
# https://renode.io

# Run firmware in simulated STM32F4
renode
(monitor) using sysbus
(monitor) mach create
(monitor) machine LoadPlatformDescription @platforms/boards/stm32f4_discovery.repl
(monitor) sysbus LoadELF @firmware.elf
(monitor) start
```

Renode simulates peripherals (UART, SPI, I2C, timers), allowing you to test firmware behavior without hardware. It supports scripted test scenarios and can run in CI.

## Static Analysis as Testing

Static analysis tools catch bugs before testing:

```bash
# cppcheck — free, catches common embedded mistakes
cppcheck --enable=all --error-exitcode=1 src/

# clang-tidy — deep analysis
clang-tidy src/adcs.c -- -mcpu=cortex-m4 -mthumb

# Valgrind for memory errors (Linux-compiled version)
valgrind --error-exitcode=1 ./test_runner
```

For MISRA-C compliance checking, see [MISRA-C](./misra-c).

## Coverage Requirements

For DO-178C DAL A/B, **MC/DC coverage** (Modified Condition/Decision Coverage) is required. Every condition in every decision must be independently shown to affect the outcome.

```c
// For this function, MC/DC requires test cases that independently exercise:
// - timeout_elapsed being true when data_valid is false
// - data_valid being true when timeout_elapsed is false
// - both false (normal operation)
bool should_use_backup_sensor(bool data_valid, bool timeout_elapsed) {
    return !data_valid || timeout_elapsed;
}
```

Tools like LDRA Testbed and VectorCAST automate MC/DC analysis and coverage measurement for certifiable software.
