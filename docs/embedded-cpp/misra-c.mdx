# MISRA-C

MISRA-C (Motor Industry Software Reliability Association C) is a set of C language coding guidelines originally written for automotive systems. It has become the dominant safety-critical C standard across aerospace, medical devices, railway, and defense.

The current version is **MISRA-C:2012** with the 2023 amendment.

---

## Why MISRA-C Exists

C is powerful precisely because it gives you low-level access to hardware and makes few guarantees about behavior. The same features that make C fast and flexible — pointer arithmetic, implicit type conversions, undefined behavior — are also the features that cause subtle, hard-to-reproduce bugs in safety-critical systems.

MISRA-C restricts the language to a well-defined, analyzable subset. Every rule has a documented rationale, usually rooted in a specific class of real bugs or undefined behavior.

---

## Rule Categories

| Category | What It Means |
|---|---|
| **Mandatory** | Must always be complied with. No deviation permitted. |
| **Required** | Must be complied with unless a formal deviation is documented and approved |
| **Advisory** | Should be complied with; deviation requires less formal justification |

## The Rules That Matter Most

### Rule 1.3 — No undefined behavior

Undefined behavior in C is not just "the result is unpredictable." It means the compiler is free to assume it never happens, and will optimize accordingly — often in ways that silently break your code.

```c
// UB: signed integer overflow
int x = INT_MAX;
int y = x + 1;   // compiler may assume this never happens → optimizes out checks

// UB: reading uninitialized variable
int val;
if (val > 0) { ... }  // val contains garbage

// UB: null pointer dereference
int* p = NULL;
*p = 5;
```

### Rule 8.7 — Functions with external linkage need declarations

Every function accessible from outside its translation unit must be declared in a header. This prevents implicit declarations and type mismatches between caller and callee.

### Rule 10.x — Essentially typed conversions

MISRA-C has a detailed type system called "essential types" that tracks whether values are signed, unsigned, Boolean, character, etc. Implicit conversions between essential types are forbidden — you must make them explicit and intentional.

```c
// Violation: implicit conversion from signed to unsigned
int16_t signed_val = -1;
uint32_t result = signed_val;  // wraps to 0xFFFFFFFF — probably not intended

// Compliant: explicit, intentional cast
uint32_t result = (uint32_t)(uint16_t)signed_val;  // clearly intentional
```

### Rule 11.x — Pointer conversions

Pointer casts are heavily restricted. Converting between pointer types or between pointers and integers must be explicit and is often flagged for review.

```c
// MISRA violation: cast from integer to pointer (required for MMIO)
volatile uint32_t* reg = (volatile uint32_t*)0x40001000;
// → Must be documented as a permitted deviation for hardware access
```

### Rule 14.4 — The controlling expression of `if` must be essentially Boolean

```c
// Violation: integer used as Boolean
int flag = get_status();
if (flag) { ... }        // flag is int, not bool — violation

// Compliant
bool flag = (get_status() != 0);
if (flag) { ... }
```

### Rule 15.5 — A function shall have a single point of exit

```c
// Violation: multiple return points
int process(int x) {
    if (x < 0) return -1;    // early return — violation
    do_work(x);
    return 0;
}

// Compliant: single exit point
int process(int x) {
    int result = 0;
    if (x < 0) {
        result = -1;
    } else {
        do_work(x);
    }
    return result;
}
```

This rule is controversial. It forces structured code but can make some functions harder to read. Many teams take a formal deviation for guard clauses.

### Rule 17.2 — No recursion

Functions shall not call themselves directly or indirectly. Recursion makes stack depth unbounded and unanalyzable.

```c
// MISRA violation
int factorial(int n) {
    if (n <= 1) return 1;
    return n * factorial(n - 1);  // recursion — banned
}

// Compliant: iterative
int factorial(int n) {
    int result = 1;
    for (int i = 2; i <= n; i++) {
        result *= i;
    }
    return result;
}
```

### Rule 21.x — Standard library restrictions

Several standard library functions are restricted or banned:

- `malloc`, `calloc`, `realloc`, `free` — **banned** (dynamic memory)
- `printf`, `scanf` — **restricted** (format string vulnerabilities, non-deterministic)
- `setjmp`, `longjmp` — **banned** (bypasses normal call stack)
- `system()` — **banned**

## Static Analysis Tools

MISRA-C compliance is verified with static analysis tools, not the compiler:

| Tool | Notes |
|---|---|
| **PC-lint Plus** | Comprehensive, commercial, widely used in safety-critical |
| **Polyspace** (MathWorks) | Deep static analysis, used in automotive and aerospace |
| **Coverity** | Commercial, also catches security vulnerabilities |
| **Cppcheck** | Open source, partial MISRA support |
| **LDRA** | Widely used in DO-178C avionics certification |

Running the compiler alone is not sufficient. MISRA violations are often valid C that compiles without warnings.

## MISRA vs. C++ (AUTOSAR, HIC++)

MISRA-C:2012 covers C. For C++, the equivalent standards are:

- **MISRA-C++:2008** — older, widely used
- **AUTOSAR C++14** — automotive, more modern, based on C++14
- **HIC++ v4.0** — High Integrity C++, used in defense and aerospace

The C++ standards follow the same philosophy: restrict the language to a deterministic, analyzable subset. Key additions for C++:

- No exceptions (`-fno-exceptions`)
- No RTTI (`-fno-rtti`)
- `virtual` functions restricted or banned in critical paths
- No `dynamic_cast`
- Templates must not increase binary size unexpectedly

## Practical Approach

In practice, most flight software teams:

1. Enable MISRA-C checking in their CI pipeline
2. Fix all mandatory violations — no exceptions
3. Fix required violations or document formal deviations
4. Apply advisory rules by judgment, especially for hardware access code
5. Maintain a deviation log that is reviewed before each release

Perfect MISRA compliance is rarely the goal. The goal is using the standard as a tool to catch the specific classes of bugs — type confusion, undefined behavior, unanalyzable control flow — that cause silent failures in safety-critical systems.
